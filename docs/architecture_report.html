Redis Clusters

Hệ thống In-memory Cache đa cụm, chịu trách nhiệm tăng tốc độ truy xuất và giảm tải cho Database chính.

📍 Khai báo & Khởi tạo

Khai báo logic: pkg/cache/redis/manager.go

Khởi tạo & Kết nối: cmd/api/setup_db.go (thông qua hàm InitDefaultManager).

🧩 Các cụm Redis (Clusters)

  • General: Dùng để lưu trữ Cache dữ liệu nghiệp vụ và lưu AES Session Keys (phục vụ mã hóa E2EE). Các Key được quản lý tập trung qua file để tránh trùng lặp.
  • Rate Limit: Cụm chuyên biệt để lưu trữ bộ đếm chống DDoS, Spam. Đứng độc lập để nếu bị ngập lụt Request cũng không làm sập bộ nhớ Cache của hệ thống chính.
  • Asynq: Cụm chỉ lưu trữ cấu hình Queue cho Worker chạy ngầm (Cronjobs, Background tasks).

⭐ Rule Tối Thượng (Golden Rule)

NOT SOURCE OF TRUTH Redis chỉ đóng vai trò bộ đệm và tăng tốc. Nếu cụm Redis mất điện, sập, hoặc bị xóa trắng Key đột ngột, hệ thống bắt buộc phải tự động truy vấn lại từ Database gốc (MongoDB/OpenSearch) để nạp lại Cache, tuyệt đối không được phép làm sai lệch hoặc mất dữ liệu kinh doanh của hệ thống.

MongoDB Clusters

Cơ sở dữ liệu NoSQL trung tâm, lưu trữ toàn bộ dữ liệu nghiệp vụ của các Tenant và System.

📍 Khai báo & Khởi tạo

Khai báo logic: pkg/database/mongodb/manager.go

Khởi tạo & Kết nối: cmd/api/setup_db.go

🧩 Các cụm MongoDB

  • General Cluster: Chứa các Collection liên quan đến dữ liệu chính của hệ thống như System, Tenant, Email Templates, Users...
  • Logging Cluster: Tách biệt hoàn toàn để chứa các dữ liệu Write-heavy (như Action Logs, Access Logs) nếu không muốn đẩy vào OpenSearch. Đảm bảo luồng ghi Log không làm nghẽn luồng đọc/ghi của nghiệp vụ chính.

OpenSearch Engine

Hệ thống lưu trữ và tìm kiếm Full-text siêu tốc, sinh ra để giải quyết bài toán Logs/Audit Trail khối lượng khổng lồ.

📍 Khai báo & Khởi tạo

Khai báo logic: pkg/database/opensearch/manager.go & time_series.go

Migration (Tạo Index/Policy): pkg/database/opensearch/migrations/init.go

⭐ Rule Tối Thượng (Golden Rule)

TIME-SERIES ISM POLICY Đối với dữ liệu dòng thời gian (Log, Metric), code Backend Tuyệt đối không được phép tự động xóa Data cũ bằng Cronjob. Hệ thống phải tạo Alias kèm theo ISM Policy (Index State Management): OpenSearch sẽ tự động Rollover đẻ Index mới khi dung lượng đạt 1GB, và tự động xóa các Index quá 30 ngày tuổi. Backend chỉ việc Ghi (Write) vào Alias.

STATIC ENTITIES Đối với các dữ liệu Business (ít cập nhật), sử dụng Alias tĩnh, Backend chủ động gọi lệnh Delete thủ công khi cần thiết.

Kafka & Asynq (Message Queue)

Trái tim của hệ thống hướng sự kiện (Event-driven) và xử lý bất đồng bộ.

📍 Khai báo & Khởi tạo

Kafka Setup: cmd/api/setup_kafka.go

Asynq Jobs: internal/system/application/jobs/

🧩 Kafka (Event Streaming)

Đóng vai trò phân phối Message giữa các Service. Hỗ trợ 3 Mode Consume:

  • Ordered: Xử lý tuần tự nghiêm ngặt (đảm bảo thứ tự sự kiện).
  • Parallel: Xử lý song song bằng Goroutine để tăng tối đa Throughput.
  • Batch: Gom lô (Ví dụ: 1000 message) rồi mới insert 1 lần vào Database.

🧩 Asynq (Cron Scheduler)

Sử dụng Redis làm Broker. Asynq chỉ làm nhiệm vụ đặt lịch (Scheduler). Khi đến giờ kích hoạt, Asynq sẽ không trực tiếp xử lý nghiệp vụ nặng mà sẽ bắn sự kiện (Push Event) sang Kafka. Kafka mới là nơi hứng và phân phối Job cho các Consumer. Quy tắc này giúp tập trung luồng xử lý Data vào một xương sống duy nhất (Kafka) và dễ giám sát.

Security Handshake (E2EE)

Luồng bắt tay bảo mật thiết lập Mã hóa Đầu - Cuối (End-to-End Encryption).

sequenceDiagram participant Client participant API as Handshake API participant Redis as Redis (General) Client->>API: 1. GET /api/v1/security/public-key API-->>Client: Trả về RSA Public Key (Server) Note over Client,API: Client tự sinh 1 chuỗi AES Key ngẫu nhiên (Session Key) Client->>Client: Mã hóa AES Key đó bằng RSA Public Key Client->>API: 2. POST /api/v1/security/handshake
{ "encrypted_key": "..." } API->>API: Dùng RSA Private Key nội bộ giải mã lấy AES Key API->>Redis: Lưu AES Key vào Redis (TTL 12h)
với Key: session_id API-->>Client: 200 OK + Trả về `session_id` Note over Client,API: Các Request nhạy cảm sau đó sẽ mã hóa Body bằng AES Key.

Mã hóa ra sao?

Luồng bảo mật này giải quyết lỗ hổng bị đánh cắp Payload ở giữa đường mạng (Man-in-the-middle). Vì AES mã hóa nhanh nên được dùng để mã hóa Data, còn RSA mã hóa chậm nên chỉ dùng để trao đổi khóa AES ban đầu.

API Request Lifecycle

Vòng đời của 1 HTTP Request từ lúc lọt vào Gateway cho tới khi lưu xuống Database.

sequenceDiagram participant Client participant RateLimit as RateLimit Middleware participant Logger as Logger Middleware participant Crypto as Payload Crypto E2EE participant Handler as HTTP Handlers participant Layer as CQRS (App/Domain) Client->>RateLimit: Gửi HTTP Request alt Vượt ngưỡng Rate Limit RateLimit-->>Client: 429 Too Many Requests else Hợp lệ RateLimit->>Logger: Bắt đầu đo thời gian (Start Timer) Logger->>Crypto: Chuyển tiếp Request alt Chứa Header X-Session-ID Crypto->>Crypto: Kéo AES Key từ Redis. Giải mã Body. end Crypto->>Handler: Truyền Plaintext JSON Handler->>Layer: Parse thành Command/Query DTO Layer->>Layer: Xử lý Business Logic / Gọi DB Layer-->>Handler: Trả về Entity / Data Handler-->>Crypto: Trả về JSON Response alt Cần mã hóa Response Crypto->>Crypto: Mã hóa field 'data' thành Base64 AES end Crypto-->>Logger: Trả về Response Logger->>Logger: Tính Latency, Đọc HTTP Status Code Logger-->>Client: Gửi Response cho Client par Đẩy Async Log Logger->>OpenSearch: Gửi Audit Log (Status, IP, Latency) qua Goroutine/Kafka end end

Chi tiết Pipeline

  • Rate Limit Middleware: Chặn ngay ngoài cổng nếu Spam. Giao tiếp với cụm Redis riêng.
  • Logger Middleware: Chặn hai đầu (Trình tự: Đo thời gian bắt đầu -> Gọi Next() -> Tính thời gian kết thúc). Không bao giờ block tiến trình chính, luôn đẩy Log đi bằng Asynchronous Goroutine.
  • Payload Crypto Middleware: Nằm sát Handler nghiệp vụ nhất, đảm nhận việc "dịch" các Data Base64 mã hóa trở lại thành JSON thô (Plaintext) để Handler chỉ việc parse bình thường mà không cần quan tâm đến bảo mật.

Backend Architecture & Conventions

Tiêu chuẩn kiến trúc, trình bày mã nguồn và các "Luật Thép" bắt buộc tuân thủ trong Sowfkun Verse.

📍 Kiến trúc 4 Lớp (Clean Architecture)

Mỗi Module (Domain) bắt buộc phải tuân theo cấu trúc Golden Standard (tham khảo internal/tenant):

  • Domain Layer: Chứa Entities kế thừa BaseEntity và Interface Repository. TUYỆT ĐỐI KHÔNG chứa logic Database hay Framework.
  • Application Layer: Phân tách CQRS (commands/ và queries/). DTO truyền vào bắt buộc kế thừa CommonCommand/Query. Tuyệt đối không gọi thẳng DB/Cache/ES ở tầng này mà phải qua Interface. Không over-engineering chia nhỏ hàm lặt vặt.
  • Infrastructure Layer: Đặt tên file theo Engine (vd: mongodb_repository.go). Thao tác Write phải gom về hàm Master để thực thi 1 lần. Mọi hàm Read bắt buộc dùng Projection (cấm SELECT *).
  • Presentation Layer: "Dumb" Controller - chỉ Parse Request, gọi UseCase và trả Standard Response. Không chứa Business Logic. Cấm dùng Dynamic Path (/:id) nếu thiết kế cho phép dùng Query Params.

🧩 Quy tắc phân vùng (Zoning Comments)

Trong các file code lớn, bắt buộc phải dùng các dải comment chuẩn để tách biệt rõ ràng cấu trúc file:

// ================= MODEL ZONE =================
// Dành cho struct DTO (Command, Query, Response).

// ================= TYPE ZONE =================
// Khai báo Interface của UseCase/Handler và private struct.

// ================= EXECUTION ZONE =================
// Hàm Constructor và hàm thực thi logic chính (Execute).

⭐ Quy tắc Nhúng (Embedding) Siêu Dữ Liệu

Thay vì khởi tạo bằng tay lặp đi lặp lại CreatedAt, TenantID..., hệ thống yêu cầu sử dụng hàm tiện ích trung tâm:

utils.SetCommonEntityData(entity, cmd.CommonCommand)

🕒 Biến môi trường & Tiêu chuẩn Thời gian

  • Naming Consistency: Biến môi trường cùng cụm hạ tầng phải chung format prefix (VD: REDIS_GENERAL_URL). Cấm hardcode thông tin nhạy cảm.
  • Global Time Standard: Mọi biến thời gian trong DB/Model/Payload bắt buộc dùng int64 Unix Timestamp (mili-giây) và ép chuẩn time.UTC(). Tuyệt đối cấm lọt field time.Time vào cấu trúc DB.
